Audience: Executive / Business Stakeholders Owner: Product Owner — Value-Added Services, Erajaya Status: Internal pitch deck (Mei 2026)

Slide 1 — Executive Summary

Apa yang kita bangun: Sebuah ERP-as-a-Service platform berbasis Odoo 19 Community Edition (CE) yang sudah ditambah 82 custom module untuk:

  1. Menutup gap fitur Odoo CE → Enterprise Edition (EE) — tanpa license fee Odoo EE.
  2. Memenuhi regulasi Indonesia (UU PDP 27/2022, Coretax DJP PER-11/PJ/2025, PSAK, BPJS, PPh, PPN, OJK ESG).
  3. Menyediakan AI Layer (Claude / OpenAI / Ollama) terintegrasi di seluruh ERP.
  4. Beroperasi sebagai multi-tenant SaaS untuk seluruh lini bisnis Erajaya & afiliasi.

Posisi strategis VAS: Platform ini adalah product enabler untuk lini Value-Added Services Erajaya — kita tidak menjual ERP, kita menjual operational backbone untuk vertikal: F&B, Active Lifestyle, Eraspace, Distribution, Service Center, Corporate Services.

Highlight angka: - 82 custom modules siap pakai - 5 lapisan arsitektur (CE core + EE-gap + Compliance + AI + Observability) - 10 tenant Y1, target 99.5% SLA - DB-per-tenant isolation - Zero EE license cost (LGPL-3 compatible)

Slide 2 — Problem Statement

Kondisi industri ERP di Indonesia:

Pain point Konsekuensi
Odoo EE license = ~USD 31/user/bulan × N user × N tenant Cost explosion saat skala
SAP / Oracle = vendor lock-in + biaya implementasi miliaran Tidak feasible untuk SMB & vertikal kecil
Localization Indonesia tidak komprehensif (Coretax, PSAK, BPJS, PDP) Tim akunting/HR manual workaround di Excel
Compliance UU PDP 27/2022 (efektif Okt 2024) jarang built-in Risiko denda 2% omzet tahunan
Coretax DJP PER-11/PJ/2025 (NSFP 17 digit) belum di-cover ERP umum Faktur ditolak, AR macet

Hipotesis VAS: Erajaya punya skala (50+ tenant calon, multi-vertikal) yang membenarkan investasi platform sendiri. ROI tercapai setelah ~tenant ke-15 dibandingkan license EE.

Slide 6 — Indonesian Regulatory Compliance

6.1 UU PDP 27/2022 (Perlindungan Data Pribadi)

Implementasi 6 modul yang men-cover seluruh kewajiban operasional UU PDP:

Pasal UU PDP Implementasi Platform
Pasal 4 — Klasifikasi data umum vs spesifik custom_pdp_core — registry pdp.field.class per ORM field
Pasal 22 — Consent explicit, granular, revocable custom_pdp_consent — pdp.consent w/ purpose taxonomy + evidence blob
Pasal 5 — Hak akses, koreksi, hapus, portabilitas custom_pdp_dsar — endpoint /pdp/dsar/request + captcha + email verif
Pasal 35 — Akuntabilitas pemrosesan custom_pdp_audit — append-only hash-chained log + PG trigger blocks UPDATE/DELETE + nightly chain-verify cron
Masking PII di view & export custom_pdp_masking — ORM read hook + jinja filter
Retention & auto-purge custom_pdp_retention — kebijakan TTL + scheduler

Differentiator: Audit log adalah immutable hash-chain (sha256 prev+payload) dengan trigger PostgreSQL BEFORE UPDATE OR DELETE yang RAISE EXCEPTION 'audit log is append-only'. Ini bukti hukum yang sulit dipertahankan dengan ERP off-the-shelf.

6.2 Coretax DJP — PER-11/PJ/2025

Kewajiban Implementasi
NSFP 17 digit (KKSSYY + 11 digit DJP) custom_coretax — field x_custom_coretax_nsfp di account.move, state machine: draft → submitted → nsfp_assigned / rejected
XML export 7 dokumen (e-Faktur Keluaran, Masukan, Bupot 21 Tetap/Bukan Tetap, 23, 26, Unifikasi) custom_coretax — wizard per template, validasi XSD lokal di data/xsd/
Bukti Potong PPh Unifikasi (22 / 23 / 4(2) / 15 / 26) custom_coretax_bupot
Sertifikat Elektronik (.p12) Fernet-encrypted storage via custom.ir.config, env-keyed
Host-to-host submission (ketika tersedia) custom_coretax_pajakku — Pajakku ASPP adapter w/ OAuth2 token cache, exponential backoff, circuit breaker, per-tenant usage meter untuk billing
Manual fallback Upload manual ke Coretax portal (jalur sah hari ini)

Strategic note: DJP belum membuka REST API B2B publik. Kita pakai adapter pattern — hari ini Pajakku ASPP, besok H2H langsung tinggal swap implementasi. Tenant tidak perlu mengganti workflow.

Slide 7 — Security Posture (DevSecOps Day-1)

7.1 Application Security

Control Implementasi
Append-only audit log PostgreSQL trigger + sha256 hash chain (custom_pdp_audit)
PII masking at ORM read custom_pdp_masking
Role-based access (RBAC) Odoo groups + record rules + ir.model.access.csv per modul
Multi-tier approval w/ delegation & OOO custom_approval_engine
HMAC signed inter-service calls Odoo → ai-gateway: X-Custom-Signature = HMAC_SHA256(secret, ts+body), drift ≤300s
Tenant allow-list per request X-Era-Tenant header validated di ai-gateway
Sertel encryption at rest Fernet (env-keyed) for .p12 files
Secrets management SOPS-encrypted in repo, decrypted at runtime
